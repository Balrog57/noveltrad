## 2024-07-13 - Path Traversal Bypass via URL Encoding in IPC Boundary
**Vulnerability:** The application was vulnerable to path traversal attacks via URL-encoded sequences (`%2e%2e%2f` instead of `../`) at the IPC boundary. The underlying mitigation relied on `path.resolve` or naive string checking (`.includes("..")`), which successfully blocked raw `..` sequences but ignored encoded equivalents.
**Learning:** Because Electron's IPC uses the Structured Clone algorithm, strings transmitted across the boundary are sent raw. Validating with low-level filesystem methods (like `path.resolve`) leaves a blind spot for URL-encoded payloads, which might be legitimately passed through the check but later decoded by deeper APIs or file handling logic. Relying solely on `decodeURIComponent` at the boundary is also flawed because it accepts valid paths that contain `%`.
**Prevention:** Always validate against path traversal *before* path resolution using a robust regular expression that explicitly captures both raw and encoded sequences, such as `/(?:\.|%2e){2}(?:$|\/|\\|%2f|%5c)/i`. Ensure this check is applied uniformly across all IPC boundaries that accept file or directory paths.
